Welcome! This document outlines how to work in this project and should be followed for any change in the repository.

## Project Overview

- **Framework**: [Astro](https://astro.build/) 6 with TypeScript (strict) and Tailwind CSS v4, deployed on Cloudflare Workers via `@astrojs/cloudflare`.
- **Node**: Astro 6 requires Node `>=22.12`. The version is pinned in `.node-version` (read by local version managers and by Cloudflare Workers Builds). When upgrading Node, bump that file â€” not just your local install â€” or the Cloudflare build will fail.
- **Content**: Blog posts and projects live in `src/content/{blog,projects}/<slug>/index.{md,mdx}` â€” one directory per entry, holding the markdown plus any co-located component files (e.g. `.tsx`). Collections are defined in `src/content.config.ts` using the Content Layer API (`glob` loaders); a `generateId` strips the trailing `/index` so each entry's `id` (and its URL) is just the directory name.
- **UI Components**: Reusable pieces are under `src/components` and `src/layouts`.
- **Pages**: Route files are in `src/pages`; dynamic routes use Astro's `[...slug].astro` convention and look entries up by `entry.id`. API endpoints go in `src/pages/api/`.
- **Styling**: Tailwind CSS v4 via the `@tailwindcss/vite` plugin. Theme config lives in `src/styles/global.css` under the `@theme` directive. Prefer Tailwind utilities for styling.

## Cloudflare

- The site builds to a Worker. Access bindings (e.g. the `blog` KV namespace) with `import { env } from "cloudflare:workers"` â€” adapter v13 removed the old `Astro.locals.runtime.env`.
- Binding types live in `worker-configuration.d.ts`, generated by `wrangler types` (committed, and prettier-ignored). Re-run `wrangler types` after editing `wrangler.jsonc`.
- `wrangler.jsonc`'s `main` must point at `@astrojs/cloudflare/entrypoints/server`, not a `dist/` path.
- `src/shims/debug.js` plus the `debug` entry in `vite.resolve.alias` (`astro.config.mjs`) are an intentional workaround: the transitive CJS `debug` dep references the `module` global, which crashes the workerd dev runtime ("module is not defined"). Leave it until fixed upstream â€” see [expressive-code#439](https://github.com/expressive-code/expressive-code/issues/439).

## Development Workflow

- Install dependencies with `npm install`.
- Run the site locally with `npm run dev` (this runs in the Cloudflare workerd runtime).
- Build for production with `npm run build`; serve the built Worker with `npm run preview`.
- Type-check with `npx astro check`.
- Format code with `npm run format` (or `npm run format:check`). Pre-commit hooks auto-format staged files; install with `pre-commit install`.

## Code Conventions

- Favor existing component patterns before introducing new ones.
- Keep imports tidy; unused imports should be removed. Never wrap imports in try/catch blocks.
- Render markdown/MDX entries with `render(entry)` from `astro:content` (the legacy `entry.render()` was removed in Astro 6).
- Prefer Tailwind utility classes for spacing, color, and typography. If you add new global styles, place them with the existing stylesheets.
- For content-driven pages, gracefully handle missing optional fields (e.g., draft flags) to avoid empty UI elements.
- Descriptions are not displayed in the UI for blog posts or projects to maintain a clean interface.

This codebase has Blog Posts and Projects. The boilerplate code to render this content should be kept in sync, unless there is an explicit reason to diverge.

## Adding New Projects

1. Create directory: `src/content/projects/project-name/`
2. Add `index.mdx` with frontmatter (title, description, date, optional draft flag)
3. Add any React components (`.tsx`) in the same directory
4. Import components in MDX: `import Component from "./Component.tsx"`
5. Use the `client:only="react"` directive for client-side-only React components
6. Create API endpoints in `src/pages/api/` if needed (server-side, with access to Cloudflare bindings via `cloudflare:workers`)

## Testing & Verification

- **Build and type-check are necessary but not sufficient.** Dev, SSR, and prerendering run in Cloudflare's workerd runtime, which behaves differently from the build â€” code can build and type-check cleanly and still return a 500 at request time.
- Before committing, actually run the app: `npm run dev`, then hit a spread of routes â€” always include `/` (homepage), a blog post, and a project page â€” and confirm each returns HTTP 200 with real content.
- Then run `npx astro check` (types) and `npm run build`.
- Note all executed commands in your final summary.
